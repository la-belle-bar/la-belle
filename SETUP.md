# La Belle — настройка безопасного деплоя (GitHub Pages + Google Apps Script)

## Как это устроено

```
Покупатель ──► GitHub Pages (статичный сайт, БЕЗ секретов)
                    │
                    ▼  fetch (каталог, заказы)
              Google Apps Script  ◄── все секреты здесь (Script Properties)
                    │
                    ▼  от имени владельца
              ПРИВАТНАЯ Google-таблица + Telegram-бот
```

Сайт на GitHub Pages полностью публичный (и код, и репозиторий), поэтому в нём
нет ни одного секрета. Всё, что требует прав, делает Google Apps Script — он
выполняется от имени владельца таблицы, а админ-операции защищены ключом
`ADMIN_TOKEN`, который проверяется на сервере при каждом запросе.

---

## Шаг 1. Сделайте таблицу приватной

Google Sheets → **Настройки доступа** → уберите «Все, у кого есть ссылка».
Доступ должен остаться только у вас. Публиковать таблицу в интернете
(«Файл → Открыть доступ → Публикация») тоже **не нужно** — если публиковали,
отмените публикацию. Сайту прямой доступ к таблице больше не требуется.

## Шаг 2. Обновите Google Apps Script

1. Откройте таблицу → **Расширения → Apps Script** (или существующий проект
   скрипта, если он уже был).
2. Замените всё содержимое `Code.gs` на файл [apps-script/Code.gs](apps-script/Code.gs)
   из этого репозитория.
3. **Project Settings (шестерёнка) → Script Properties** — добавьте свойства:

   | Свойство | Значение |
   |---|---|
   | `SPREADSHEET_ID` | `1Jyrtv0BnD03CBCqCbfF1puSiAVTqG4p-MPSaXCYFOgM` |
   | `CATALOG_SHEET_GID` | `1821735406` |
   | `ORDERS_SHEET_GID` | `926886605` |
   | `PROMO_SHEET_GID` | *(необязательно, для промокодов)* gid листа `promocodes` |
   | `ADMIN_TOKEN` | длинный случайный ключ, см. ниже |
   | `TELEGRAM_BOT_TOKEN` | *(необязательно)* токен бота от @BotFather |
   | `TELEGRAM_CHAT_ID` | *(необязательно)* id чата для уведомлений |

   `SPREADSHEET_ID` — из URL таблицы (`.../spreadsheets/d/<ВОТ_ЭТО>/edit`),
   gid листа — число после `#gid=` при открытом нужном листе.

4. Сгенерируйте `ADMIN_TOKEN` (это ваш «пароль» от админки — 32+ случайных
   символов). В PowerShell:

   ```powershell
   -join ((48..57)+(65..90)+(97..122) | Get-Random -Count 40 | ForEach-Object {[char]$_})
   ```

   Никому его не показывайте и **никогда не вставляйте в код сайта**.

## Шаг 3. Задеплойте веб-приложение

1. **Deploy → New deployment → Web app** (тип: Web app).
2. **Execute as: Me** (важно — скрипт работает от вашего имени).
3. **Who has access: Anyone** (это открывает только API скрипта, не таблицу).
4. Скопируйте URL вида `https://script.google.com/macros/s/…/exec`.

Если деплой уже существовал, обновите его: **Deploy → Manage deployments →
✏️ Edit → Version: New version → Deploy** — тогда URL не изменится.

## Шаг 4. Пропишите URL в сайте

В [assets/js/config.js](assets/js/config.js) укажите ваш URL в `apiUrl`.
Это единственная настройка на стороне сайта. Секретов в этом файле быть
не должно.

## Шаг 5. Telegram-уведомления (по желанию)

1. Создайте бота у [@BotFather](https://t.me/BotFather) → получите токен.
2. Добавьте бота в нужный чат/группу (или напишите ему в личку).
3. Узнайте chat id: откройте `https://api.telegram.org/bot<ТОКЕН>/getUpdates`
   после отправки сообщения в чат — в ответе будет `"chat":{"id":…}`.
4. Впишите `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID` в Script Properties.

Уведомления отправляет сервер при приёме заказа — токен бота в браузер
не попадает.

## Шаг 6. GitHub Pages

1. Repository → **Settings → Pages** → включите **Enforce HTTPS**.
2. Помните: на бесплатном тарифе репозиторий для Pages публичный. Это
   нормально — в коде секретов нет. Но **не коммитьте** реальные токены даже
   «на минутку»: git-история публична навсегда. Если токен попал в историю —
   просто смените `ADMIN_TOKEN` в Script Properties.

## Как пользоваться админкой

Откройте `admin.html`, введите `ADMIN_TOKEN`. Ключ проверяется сервером и
хранится только в текущей вкладке (sessionStorage) до выхода/закрытия.
Всё, что делает админка (заказы, статусы, правка товаров), Apps Script
выполняет только при верном ключе.

## Что защищено и от чего

- **Данные клиентов** (телефоны, адреса) лежат в приватной таблице и отдаются
  только по админ-ключу. Публичного CSV-экспорта больше нет.
- **Правка каталога и статусов** — только с ключом, проверка на сервере.
- **Подмена цены в браузере** — сервер сверяет цены заказа с каталогом и пишет
  результат в колонку `price_check` листа заказов: заказ с подменённой ценой
  будет помечен `MISMATCH!` (и выделен в Telegram-уведомлении). Проверяйте её
  до подтверждения заказа.
- **Спам заказами** — скрытое honeypot-поле против ботов + общий лимит
  30 заказов / 10 минут на сервере.
- **XSS** — строгая Content-Security-Policy на всех страницах + экранирование
  HTML во всём рендеринге.

## Промокоды и сертификаты (по желанию)

Скидки применяются в корзине и **пересчитываются на сервере** при приёме
заказа — присланной браузером сумме и скидке доверия нет.

### Шаг A. Лист `promocodes`

Создайте в той же таблице новый лист (например `promocodes`) с такими
колонками в первой строке (регистр и порядок не важны, важны имена):

| Колонка | Назначение |
|---|---|
| `code` | код, который вводит покупатель (напр. `WELCOME10`) |
| `type` | `percent` (процент), `fixed` (фикс. сумма ₸) или `cert` (сертификат) |
| `value` | число: для `percent` — 0–100, для `fixed`/`cert` — сумма в ₸ |
| `min_order` | *(необяз.)* минимальная сумма заказа для применения |
| `active` | `TRUE`/пусто — активен; `FALSE`/`нет`/`0` — выключен |
| `expires_at` | *(необяз.)* дата окончания (напр. `2026-12-31`) |
| `usage_limit` | *(необяз.)* сколько раз всего можно применить (для сертификата — `1`) |
| `used_count` | счётчик использований; сервер увеличивает сам, начните с `0`/пусто |
| `note` | *(необяз.)* заметка для себя |

Возьмите gid этого листа (число после `#gid=` в URL) и добавьте свойство
`PROMO_SHEET_GID` в **Script Properties**. Новые коды добавляются прямо в
таблицу — отдельной админки не требуется.

### Шаг B. Колонки скидки в листе заказов

В лист заказов добавьте в строку заголовков три колонки: `subtotal`,
`discount`, `promo_code`. Тогда сервер запишет туда подытог, скидку и
применённый код. Если их не добавить, заказы всё равно принимаются — просто
без этих полей (в `total` уже лежит сумма со скидкой).

После правок обновите деплой: **Deploy → Manage deployments → Edit →
New version → Deploy** (URL сохранится).

### Что защищено

- Сервер сам считает подытог из позиций, сверяет промокод с листом
  (активность, срок, лимит, минимальную сумму) и пишет итог со скидкой —
  подделать скидку из браузера нельзя.
- Использование сертификата/лимитного кода списывается под тем же локом, что и
  запись заказа.

### Покупка сертификатов

На главной есть блок «Купить сертификат»: покупатель выбирает номинал, кладёт
сертификат в корзину и оформляет как обычный заказ. Код **выдаётся только когда
вы отметите заказ «Оплачен»** в админке (не раньше — иначе код можно было бы
получить без оплаты).

Что нужно:
- Лист `promocodes` (тот же, что для промокодов) — сгенерированный `cert`-код
  туда пишется автоматически (type=cert, одноразовый).
- В лист заказов добавьте колонку `cert_codes` — туда сервер запишет выданные
  коды (и не выдаст повторно при повторной отметке «Оплачен»).

Когда вы переводите заказ в статус «Оплачен», админка покажет выданные коды
(«Выданы сертификаты: GIFT-XXXX…») — их можно отправить покупателю. Номиналы на
витрине настраиваются в `config.js` полем `certificateAmounts` (по умолчанию
5000/10000/15000/25000).

## Отзывы и рейтинги (по желанию)

Покупатели оставляют отзыв на странице аромата. Отзыв **не публикуется сразу** —
он попадает в лист со статусом `pending` и появляется на сайте только после того,
как вы вручную поставите `approved` (модерация).

### Шаг A. Лист `reviews`

Создайте лист (например `reviews`) с колонками в первой строке:

| Колонка | Назначение |
|---|---|
| `timestamp` | дата отзыва (заполняет сервер) |
| `review_id` | id отзыва (заполняет сервер) |
| `product_key` | к какому аромату (заполняет сервер: `sku:…` или `name:бренд\|название`) |
| `name` | имя автора |
| `rating` | оценка 1–5 |
| `text` | текст отзыва |
| `status` | `pending` (новый) → поставьте `approved`, чтобы опубликовать; `rejected` скрыть |

Добавьте gid листа в **Script Properties** как `REVIEW_SHEET_GID` и обновите
деплой (New version).

### Как модерировать

Открывайте лист `reviews`, читайте новые строки (`pending`) и меняйте `status`
на `approved` — отзыв появится на сайте и учтётся в среднем рейтинге аромата.
Спам просто оставляйте `pending` или ставьте `rejected`.

### Что защищено

- Публично отдаются **только** отзывы со статусом `approved`.
- Отправка отзыва защищена honeypot-полем и лимитом 20 отзывов / 10 минут.
- Весь текст экранируется при выводе (строгая CSP + экранирование HTML).

## Бонусная программа (лояльность по телефону, по желанию)

Клиент опознаётся по **телефону**. Баллы начисляются и списываются **при
отметке заказа «Оплачен»** (не при оформлении — так неоплаченный заказ не сжигает
баллы). Списанные баллы уменьшают сумму как скидка; сервер сверяет баланс по
телефону и пересчитывает итог — подделать из браузера нельзя.

Что нужно:
1. Создайте лист `customers` с колонками:
   `phone, name, points, total_spent, orders_count, first_order, last_order`.
   Добавьте его gid в Script Properties как `CUSTOMER_SHEET_GID`.
2. В лист заказов добавьте две колонки: `points_redeemed` и `loyalty_earned`
   (сервер пишет туда списанные/начисленные баллы; `loyalty_earned` служит
   отметкой, чтобы не начислить дважды).
3. Процент начисления настраивается **в админке** (вкладка «Настройки»,
   только владелец) — сервер хранит его в свойстве `LOYALTY_EARN_PERCENT`
   (по умолчанию `5`). Задавать свойство вручную не обязательно. 1 балл = 1 ₸.
4. Обновите деплой (New version).

Как это работает для покупателя: в корзине блок «Бонусные баллы» → «Проверить»
(по телефону из формы) → показывает баланс → можно списать баллы (до суммы
заказа). Баллы за заказ появятся после того, как вы отметите его «Оплачен».

Баланс по телефону отдаётся публично, но **только число баллов** и с лимитом
60 проверок / 10 минут — перечислять номера бессмысленно.

Отмена оплаченного заказа **автоматически возвращает** списанные баллы и
откатывает начисленные (счётчики клиента не уходят в минус). Промокод **не
действует на подарочные сертификаты** (они эквивалент денег). База лояльности
видна в админке на вкладке «Клиенты».

**Осознанное ограничение:** баланс виден любому, кто знает номер (личный кабинет
из следующего раздела это частично закрывает; строгий OTP — по желанию позже).

## Личный кабинет клиента (по желанию)

Страница `account.html` — вход по **телефону + номеру любого своего заказа**
(номер клиент видит в подтверждении). После входа показываются баланс баллов,
кэшбэк, количество заказов, потраченная сумма и **история заказов**.

Как это устроено:
- Вход сверяется на сервере: заказ с таким номером и телефоном должен
  существовать в листе заказов. Совпало — выдаётся **сессия** (подписанный токен
  на 30 дней), которая хранится в браузере клиента.
- Токен подписывается секретом `SESSION_SECRET` — он **генерируется
  автоматически** при первом входе и сохраняется в Script Properties. Задавать
  его вручную не нужно (но можно — любой длинной случайной строкой).
- Историю заказов сервер отдаёт только по валидному токену и **только по
  своему** телефону.

Отдельных листов не требуется — используются уже существующие заказы и
`customers`. Достаточно передеплоя. Если хотите убрать кабинет — уберите ссылку
«Кабинет» из меню; страница ничего не ломает.

**Осознанное ограничение:** это «вход по знанию» (телефон + номер заказа), а не
строгий OTP. Для магазина такого размера этого достаточно; при желании поверх
можно добавить email- или SMS-код.

## Email покупателя (только сбор, писем не шлём)

В чекауте есть необязательное поле email. Оно сохраняется в заказ (**для базы
клиентов**), но автоматических писем сайт **не отправляет** — уведомления идут в
Telegram (вам) и через кнопку WhatsApp (клиент пишет вам сам).

Чтобы email попадал в таблицу, добавьте в лист заказов колонку `email`. Если
колонки нет — заказы всё равно принимаются, просто без email.

Разрешение на отправку почты (scope Gmail) больше не нужно.

## Роли админов, аналитика и журнал (по желанию)

### Роли: владелец и менеджер

Раньше был один `ADMIN_TOKEN`. Теперь можно завести второй токен для менеджера:

| Свойство | Роль | Доступ |
|---|---|---|
| `ADMIN_TOKEN` (или `ADMIN_TOKEN_OWNER`) | владелец | всё: заказы, товары, отзывы, **аналитика, журнал** |
| `ADMIN_TOKEN_MANAGER` | менеджер | заказы, товары, отзывы (без аналитики и журнала) |

Старый `ADMIN_TOKEN` продолжает работать как владелец — ничего переделывать не
нужно. Менеджерский токен добавьте только если он вам нужен. Роль определяется
сервером по токену; вкладки «Аналитика» и «Журнал» видны только владельцу.

Вход в админку — тот же: вводите токен, сервер возвращает роль, интерфейс
подстраивается.

### Журнал изменений

Создайте лист (например `log`) с колонками `timestamp, actor_role, action,
target, details` и добавьте его gid как `LOG_SHEET_GID`. Тогда каждое изменение
из админки (статус заказа, правка товара, модерация отзыва) пишется строкой:
кто (роль), что, над чем. Вкладка «Журнал» (владелец) показывает последние
записи. Если лист не создавать — операции всё равно работают, просто без
журнала.

### Аналитика

Вкладка «Аналитика» (владелец) считается прямо в браузере из уже загруженных
заказов — новых листов и настроек не требует. Показывает выручку (оплаченные и
завершённые), средний чек, конверсию в оплату и топ ароматов по количеству.

### Модерация отзывов в админке

Вкладка «Отзывы» (владелец и менеджер) показывает все отзывы, включая новые.
Кнопки «Одобрить»/«Отклонить» меняют статус на сервере — редактировать таблицу
вручную больше не обязательно.

После добавления свойств обновите деплой (**Manage deployments → Edit → New
version → Deploy**).

## Ограничения (осознанные)

- Лимит Apps Script для личного аккаунта — до ~20 000 запросов/день к веб-апп.
  Для небольшого магазина это с запасом.
- «Вход» в админку — это проверка ключа сервером; сам файл admin.html публичен
  (это просто интерфейс). Без ключа он не может ни прочитать, ни изменить
  ничего.
- Заказ, оформленный покупателем, дублируется в localStorage его браузера —
  это только его локальная история, другим она не видна.
